## 三、分 JD 匹配度评估

> 8 类 JD 可归为三类：**推理引擎架构岗**、**算子/Kernel 岗**、**推理服务/系统岗**。

### JD 1 & 2：通用「大模型推理引擎研发」

**匹配度：约 65%** — 国产平台向岗位里较合适

| 要求 | 匹配 | 说明 |
|------|------|------|
| 本科+、3 年+、百亿参数 | ✅ | 学历、年限、20+ 大模型均满足 |
| 模型量化 | ✅✅ | 核心强项 |
| 算子优化 | ✅ | Ascend C + 量化算子 |
| 显存管理 | ⚠️ | 量化降显存有，但未单独表述 |
| CANN/国产平台 | ✅✅ | 明显加分项 |
| Ascend C / 自定义算子 | ✅ | 算子调试 + 生态经验 |
| TensorRT/vLLM/SGLang | ❌ | 主要缺口 |
| FlashAttention | ❌ | 需补认知或经验 |
| CUDA | ❌ | 无 GPU 侧经验 |
| 分布式推理通信 | ⚠️ | 多卡调试 ≠ 分布式推理设计 |
| 计算图编译/算子融合/动态批处理 | ❌ | 推理引擎核心模块未直接体现 |

**结论：** 投 **强调 CANN/国产芯片优先** 的版本较稳；若硬性要求 TensorRT + CUDA，简历关可能难过，需靠面试讲「方法论可迁移」。

---

### JD 3：自研硬件 + 编译/通信/低比特 + PD 分离 + KV Cache

**匹配度：约 55%**

| 要求 | 匹配 |
|------|------|
| 低比特计算 | ✅✅ MXFP/W4A4 完全对口 |
| 推理引擎架构设计 | ⚠️ 协同引擎，非主导架构 |
| 编译优化 | ❌ |
| 通信优化 | ⚠️ 多卡调试有，非推理通信调度 |
| PD 分离 / KV Cache / 投机采样 | ❌ |
| 多维并行（张量/流水/专家） | ❌ |
| 跨部门协同 | ✅ |

**结论：** **低比特 + 自研芯片** 方向匹配；**推理引擎架构 + 分布式调度** 是主要短板。适合投「量化/低比特推理优化」子方向，而非「引擎架构 Owner」。

---

### JD 4：高性能算子 / Kernel / 编译器

**匹配度：约 70%** — 8 个里相对最对口之一

| 要求 | 匹配 |
|------|------|
| CANN / Ascend C | ✅✅ |
| 手写高性能算子 | ⚠️ 量化算子有，缺「Kernel 性能倍数」表述 |
| 特定芯片深度调优（华为） | ✅✅ |
| GPU/NPU 异构 | ✅ NPU 侧强，GPU 弱 |
| MLIR/LLVM/Triton/TVM | ❌ |
| 编译器技术 | ⚠️ 熟悉 CANN 编译链，非 MLIR 开发 |

**结论：** 若岗位偏 **算子/Kernel/芯片侧优化**，比纯推理框架岗更匹配。建议在简历里把「量化算子实现」写得更像 **性能 Kernel 案例**（例如算子加速比、访存优化手段）。

---

### JD 5：PD 分离 + vLLM/SGLang + 并行 + 图优化

**匹配度：约 45%** — 8 个里偏难

| 要求 | 匹配 |
|------|------|
| vLLM/SGLang/TensorRT-LLM | ❌ |
| 数据/流水线并行 | ❌ |
| NVLINK/GPU RDMA | ❌ |
| torch.compile/Triton 图优化 | ❌ |
| CUDA/Ascend C | ⚠️ 仅 Ascend C |
| 模型分析优化 | ✅ Qwen/DeepSeek/Wan 适配 |
| 量化 | ✅ |

**结论：** 典型 **GPU 推理框架 + 分布式系统** 岗，当前简历匹配度最低。除非 JD 写明「Ascend C 优先」，否则建议低优先级，或先补 vLLM 认知再投。

---

### JD 6：推理服务框架 + KV Cache + 分布式缓存 + 微服务

**匹配度：约 40%**

| 要求 | 匹配 |
|------|------|
| vLLM/SGLang | ❌ |
| KV Cache 显存/内存/存储 | ❌ |
| 分布式系统/微服务/容器 | ❌ |
| 推理服务调度/生命周期 | ❌ |
| 量化工具链 | ✅ Model-free 转换、量化工具 |
| C++/Python 模块化设计 | ⚠️ 有基础，缺服务化证据 |

**结论：** 偏 **在线推理服务 / AI Infra 平台**，背景更偏 **离线量化 + 芯片侧**，重合度低。

---

### JD 7：多模态推理 + Continuous Batching + FP8 量化

**匹配度：约 50%**

| 要求 | 匹配 |
|------|------|
| FP8/INT8 量化 | ✅ MXFP8 直接相关 |
| 多模态（VL/Omni） | ⚠️ Wan2.2 是视频生成，非 VL 理解 |
| KV Cache / Continuous Batching | ❌ |
| PyTorch + Profiling | ⚠️ 有 Pipeline，缺 profiling 表述 |
| vLLM/SGLang/Qwen-VL 优化 | ❌ |
| Transformer 理解 | ⚠️ implicit，未写 |

**结论：** **FP8 量化** 是亮点；多模态和服务侧优化是缺口。若岗位强调「量化 + 多模态模型部署」，Wan2.2 可再包装成多模态推理经验。

---

### JD 8 & 9：AI Infra + TensorRT-LLM/vLLM + FlashAttn + 投机解码

**匹配度：约 50%**

| 要求 | 匹配 |
|------|------|
| 量化/剪枝/蒸馏 | ✅ 量化强，剪枝/蒸馏未写 |
| GPTQ/AWQ | ⚠️ 写了「熟悉范式」，非自研 |
| TensorRT-LLM/vLLM/TGI | ❌ |
| FlashAttention/PageAttn/GQA | ❌ |
| Speculative Decoding | ❌ |
| CUDA/cutlass/cublas | ❌ |
| 分布式系统 | ⚠️ 弱 |
| 异构计算优化 | ✅ NPU 侧 |
| MoE/MLA/Diffusion | ⚠️ Wan2.2 涉及 Diffusion |

**结论：** **量化 + 成本优化** 叙事能打动人；**GPU 推理框架 + Attention 优化** 是硬缺口。

## 七、一句话总结

**简历在「大模型量化 + 昇腾 CANN + 0-day 落地 + 算子/芯片侧优化」上，对这 8 类 JD 属于强匹配；在「GPU 推理框架（vLLM/SGLang）、KV Cache/批处理调度、分布式推理架构、CUDA 生态」上，属于系统性缺口。**

**最稳策略：** 优先投 **国产芯片 / 低比特量化 / 算子优化** 向岗位；投 GPU 框架岗时，用「量化 + 引擎协同 + 可迁移方法论」打，并提前备好 vLLM/FlashAttn 的理论问答。
